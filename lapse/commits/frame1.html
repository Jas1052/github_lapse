 1# Acai 
12
 3Disclaimer: This is not an official Google product. 
 4 
 5Acai makes it easy to write functional tests of your application 
 6with JUnit4 and Guice. 
 7 
 8Acai makes it simple to: 
 9- Inject the helper classes you need into tests 
 10- Start any services needed by your tests 
 11- Run between-test cleanup of these services 
 12- Start up multiple services for testing in the right order 
 13 
 14Acai is designed for large functional tests of your application. For 
 15example it can help with writing tests which start your backend and frontend 
 16server in a self-contained mode with their dependencies faked out and then 
 17validates some key user scenarios with Webdriver to give you confidence your 
 18complete system works correctly. It can also be useful for tests which validate 
 19the integration of a small set of components. Note however that for smaller 
 20unit-tests we generally recommend you create the class under test manually 
 21rather than using Acai. 
 22 
 23## Using Acai to inject a test 
 24The simplest test using Acai doesn't register any TestingService bindings 
 25at all, it just uses Acai to inject a test with a module: 
 26 
 27```Java 
 28@RunWith(JUnit4.class) 
 29public class SimpleTest { 
 30 @Rule public Acai acai = new Acai(MyTestModule.class); 
 31 
 32 @Inject private MyClass foo; 
 33 
 34 @Test 
 35 public void checkSomethingWorks() { 
 36   // Use the injected value of foo here 
 37 } 
 38 
 39 private static class MyTestModule extends AbstractModule { 
 40   @Override protected void configure() { 
 41     bind(MyClass.class).to(MyClassImpl.class); 
 42   } 
 43 } 
 44} 
 45```

 47## Using Acai to start services 
 48The real power of Acai comes when your production server is configured 
 49with Guice and you create an alternate test module which configures your server 
 50with heavyweight dependencies like databases replaced with local in-memory 
 51implementations. You could then start this server once for all tests in the 
 52suite (to avoid waiting for it to start between each test) and wipe the 
 53database between tests (to cheaply isolate test-cases from one-another). 
 54 
 55The following example shows how this pattern would be used in tests: 
 56 
 57```Java 
 58@RunWith(JUnit4.class) 
 59public class ExampleFunctionalTest { 
 60 @Rule public Acai acai = new Acai(MyTestModule.class); 
 61 
 62 @Inject private MyServerClient serverClient; 
 63 
 64 @Test 
 65 public void checkSomethingWorks() { 
 66   // Call the running server and test some behaviour here. 
 67   // Any state will be cleared by MyFakeDatabaseWiper after each 
 68   // test case. 
 69 } 
 70 
 71 private static class MyTestModule extends AbstractModule { 
 72   @Override protected void configure() { 
 73     // Normal Guice modules which configure your 
 74     // server with in-memory versions of backends. 
 75     install(MyServerModule()); 
 76     install(MyFakeDatabaseModule()); 
 77 
 78     install(new TestingServiceModule() { 
 79       @Override protected void configureTestingServices() { 
 80         bindTestingService(MyServerRunner.class); 
 81         bindTestingService(MyFakeDatabaseWiper.class); 
 82       } 
 83     }); 
 84   } 
 85 } 
 86 
 87 private static class MyServerRunner implements TestingService { 
 88   @Inject private MyServer myServer; 
 89 
 90   @BeforeSuite public void startServer() { 
 91     myServer.start().awaitStarted(); 
 92   } 
 93 } 
 94 
 95 private static class MyFakeDatabaseWiper implements TestingService { 
 96   @Inject private MyFakeDatabse myFakeDatabase; 
 97 
 98   @AfterTest public void wipeDatabase() { 
 99     myFakeDatabase.wipe(); 
 100   } 
 101 } 
 102} 
 103``` 
 104 
 105Note that when a module is passed to Acai in a rule any @BeforeSuite 
 106methods are only executed once per suite even if the same module is used in 
 107multiple Acai rules in multiple different test classes within that suite. 
 108This allows tests of the server to be structured into test classes according to 
 109the functionality being tested. 
 110 
 111## Services which depend upon each other 
 112If the services you need to start for tests must be started in a specific order 
 113you can express this using the `@DependsOn` annotation. 
 114 
 115For example: 
 116 
 117```Java 
 118@RunWith(JUnit4.class) 
 119public class ExampleFrontendWebdriverTest { 
 120 @Rule public Acai acai = new Acai(MyTestModule.class); 
 121 
 122 @Inject private SomeFrontendFeaturePageObject featurePage; 
 123 
 124 @Test 
 125 public void checkSomethingWorks() { 
 126   // Test the frontend client using the webdriver page 
 127   // object here. 
 128 } 
 129 
 130 private static class MyTestModule extends AbstractModule { 
 131   @Override protected void configure() { 
 132     // Normal Guice modules which configure your 
 133     // server with in-memory versions of servers and 
 134     // a test module configuring a webdriver client. 
 135     install(MyServerModule()); 
 136     install(MyFakeDatabaseModule()); 
 137     install(WebDriverModule()); 
 138 
 139     install(new TestingServiceModule() { 
 140       @Override protected void configureTestingServices() { 
 141         bindTestingService(MyServerRunner.class); 
 142         bindTestingService(MyBackendRunner.class); 
 143       } 
 144     }); 
 145   } 
 146 } 
 147 
 148 @DependsOn(MyBackendRunner.class) 
 149 private static class MyFrontendRunner implements TestingService { 
 150   @Inject private MyFrontendServer myFrontendServer; 
 151 
 152   @BeforeSuite public void startServer() { 
 153     myFrontendServer.start().awaitStarted(); 
 154   } 
 155 } 
 156 
 157 private static class MyBackendRunner implements TestingService { 
 158   @Inject private MyBackendServer myBackendServer; 
 159 
 160   @BeforeSuite public void startServer() { 
 161     myBackendServer.start().awaitStarted(); 
 162   } 
 163 } 
 164} 
 165``` 
 166 
 167In the above example `MyFrontendRunner` is annotated 
 168`@DependsOn(MyBackendRunner.class)` which will cause Acai to start the 
 169backend server before starting the frontend.

 171## API 
 172As shown in the above examples Acai has a very small API surface. 
 173Firstly, and most importantly, there is the `Acai` rule class itself 
 174which is used as a JUnit4 `@Rule` and is passed a module class to be used to 
 175configure the test. 
 176 
 177The module class passed to the `Acai` constructor may optionally use 
 178`TestingServiceModule` to bind one or more `TestingService` implementations. 
 179 
 180The `TestingService` interface is purely a marker to allow Acai to know 
 181which classes provide testing services. To actually do anything implementations 
 182of this interface should add zero argument methods annotated with one of 
 183`@BeforeSuite`, `@BeforeTest` or `@AfterTest`. These methods will be run before 
 184the suite, before each test or after each test respectively. You may add as 
 185many methods annotated with these annotations as you wish to a 
 186`TestingService`; Acai will find and run them all when appropriate. 
 187 
 188Finally a `TestingService` implementation can be annotated `@DependsOn` to 
 189signal its `@BeforeSuite` and `@BeforeTest` methods need to be run after 
 190those of another `TestingService`. This provides a simple declarative mechanism 
 191to order service startup in tests. 
 192 
 193Refer to the examples above to see the API in action.
